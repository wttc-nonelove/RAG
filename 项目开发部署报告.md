# RAG 智能知识库问答系统项目开发部署报告

## 一、项目概述

### 1.1 项目背景

随着企业内部制度、技术文档、政策法规和业务资料数量不断增加，传统关键词检索难以满足用户对“准确问答、跨文档综合、多轮追问、来源可追溯”的使用需求。本项目围绕私有知识库场景，设计并实现了一套基于 RAG（Retrieval-Augmented Generation，检索增强生成）架构的智能知识库问答系统。

系统允许管理员上传 PDF、DOCX、TXT、MD、Excel、CSV 等文件，后台自动完成文本抽取、清洗分块、Embedding 向量化、ChromaDB 向量入库、Neo4j 知识图谱构建，并在用户提问时结合向量检索、知识图谱路径、对话历史和大语言模型生成回答。回答结果可展示引用来源、相关片段、知识图谱证据和模型调用消耗，提升了问答的可信度与可管理性。

### 1.2 项目目标

本项目的总体目标是建设一个可本地部署、支持私有文档接入、具备检索增强生成能力的智能问答平台。具体目标如下：

1. 实现用户登录、权限控制、普通用户问答和管理员后台管理。
2. 实现多格式文档上传、解析、预览、删除、重新解析和版本化管理。
3. 构建基于 ChromaDB 的向量检索能力，支持知识片段召回和相似度筛选。
4. 构建基于 Neo4j 的知识图谱能力，支持实体关系抽取、图谱可视化、关系路径检索和人工维护。
5. 实现多轮对话、模型选择、Prompt 配置、Token 统计和问答历史管理。
6. 使用 Docker Compose 统一编排前端、后端、数据库、向量库、图数据库和缓存服务，降低部署复杂度。

### 1.3 系统使用对象

系统主要面向两类用户：

| 用户角色 | 主要职责 |
|---|---|
| 管理员 | 维护用户、上传知识库文档、管理模型供应商、调整系统参数、查看问答历史和系统运行统计 |
| 普通用户 | 基于已导入知识库进行智能问答、查看个人问答历史、维护个人资料 |

## 二、开发环境与技术选型

### 2.1 技术架构

项目采用前后端分离架构，后端对外提供 REST API，前端通过 Axios 调用接口完成业务交互。数据侧使用 MySQL 存储结构化业务数据，ChromaDB 存储文本向量，Neo4j 存储知识图谱，Redis 作为缓存和扩展任务队列基础组件。

| 层级 | 技术选型 | 作用 |
|---|---|---|
| 前端 | Vue 3、Vue Router、Pinia、Element Plus、ECharts、Vite | 页面渲染、路由控制、状态管理、后台管理和图谱可视化 |
| 后端 | Python 3.11、FastAPI、SQLAlchemy Async、Pydantic | API 服务、业务逻辑、数据访问、参数校验 |
| 关系数据库 | MySQL 8.0 | 用户、文档、会话、消息、模型、配置、Token 统计等业务数据 |
| 向量数据库 | ChromaDB | 文档分块向量、相似度检索 |
| 图数据库 | Neo4j 4.4 | 实体、关系、文档结构、知识路径 |
| 缓存组件 | Redis 7 | 缓存和后续异步任务扩展 |
| 大模型接口 | DeepSeek API，兼容 OpenAI 风格接口 | 对话生成、Embedding、实体关系抽取 |
| 部署方式 | Docker、Docker Compose | 多服务容器化部署 |

### 3.2 非功能性需求

1. 可部署性：系统应能够通过 Docker Compose 在本地或服务器环境中一键启动。
2. 可扩展性：模型供应商和模型配置应支持动态维护，便于后续接入 OpenAI 兼容接口。
3. 可维护性：后端按照 router、service、dao、model 分层组织，前端按照 view、component、api、store 分层组织。
4. 安全性：用户密码使用 bcrypt 哈希存储，API Key 使用 AES 加密存储，接口通过 JWT 鉴权。
5. 可追溯性：问答结果保留引用来源、图谱证据、使用模型、Token 消耗和响应时间。
6. 容错性：文档解析失败记录错误信息，Embedding 远程调用失败时具备本地哈希向量回退机制。

## 四、系统总体设计

### 4.1 总体架构

系统采用“前端界面 + 后端 API + 多数据存储 + 大模型服务”的组合架构。

```text
用户浏览器
   │
   ▼
Vue 3 前端服务（rag-frontend:15173）
   │  REST API
   ▼
FastAPI 后端服务（rag-backend:18080）
   ├── MySQL：用户、文档、会话、模型、配置、统计
   ├── ChromaDB：文档分块向量检索
   ├── Neo4j：实体关系、文档结构、关系路径
   ├── Redis：缓存与任务扩展
   └── DeepSeek / OpenAI 兼容 API：对话、Embedding、知识抽取
```

### 4.2 后端分层设计

后端入口文件为 `app/main.py`，注册认证、文档、问答、图谱、用户、历史、工作台、配置、模型等路由。核心分层如下：

| 层级 | 目录 | 职责 |
|---|---|---|
| 路由层 | `routers/` | 接收 HTTP 请求、鉴权、参数校验、组织响应 |
| 服务层 | `services/` | 实现业务流程，如 RAG 问答、文档解析、图谱检索 |
| 数据访问层 | `dao/` | 封装 MySQL、ChromaDB、Neo4j 的读写操作 |
| 模型层 | `models/` | 定义 SQLAlchemy ORM 模型 |
| 工具层 | `utils/` | LLM 客户端、Embedding、文本分块、加密、日志等通用能力 |

### 4.3 前端设计

前端使用 Vue 3 + Element Plus 构建管理后台和用户端页面。路由通过 `src/router/index.js` 配置，登录页不需要鉴权，其他页面需要 Token；管理员页面增加 `role: admin` 元信息，普通用户访问管理员页面时自动跳转到问答页面。

## 五、核心模块开发实现

### 5.1 用户认证与权限控制

系统使用 JWT 实现接口鉴权。用户登录后，后端生成 Token，前端将 Token 和用户信息保存到本地存储中，后续请求通过 Axios 拦截器携带认证信息。后端依赖项 `get_current_user` 负责解析 Token 并读取当前用户；管理员接口通过角色字段控制访问权限。

用户密码不以明文保存，初始化管理员账号在 `init.sql` 中写入 bcrypt 哈希值，默认管理员账号为 `admin / admin123`。生产环境部署时应及时修改默认密码。

### 5.2 文档管理与解析流程

文档管理模块是系统知识库构建的入口。管理员上传文件后，系统先保存原始文件并在 MySQL `documents` 表中创建记录，状态为 `pending`，随后进入解析流程。

解析流程如下：

```text
上传文件
  → 保存文件
  → 创建 documents 记录
  → 提取文本
  → 文本清洗
  → 按配置分块并记录章节路径、前后片段 ID
  → 调用 Embedding 模型生成向量
  → 写入 ChromaDB
  → 构建文档结构图
  → 调用 LLM 抽取实体关系
  → 写入 Neo4j
  → 更新文档状态为 completed
```

系统支持 PDF、DOCX、TXT、MD、XLSX、XLS、CSV 等格式。其中 PDF 通过 `pdfplumber` 抽取文本，DOCX 通过 `python-docx` 抽取段落，Excel 通过 `openpyxl` 读取工作表内容。解析失败时，文档状态更新为 `failed` 并保存错误信息，管理员可在知识库页面重新解析。

### 5.3 RAG 智能问答流程

用户提问后，问答接口 `/api/v1/qa/ask` 负责创建或复用会话、写入用户消息、解析默认模型，并调用 `qa_service.rag_pipeline()` 执行 RAG 流程。

RAG 流程包括：

1. 读取系统配置，包括 `top_k`、`similarity_threshold`、`history_rounds`、`temperature`、`top_p`、`max_tokens` 等参数。
2. 保存用户问题，并读取最近若干轮对话历史。
3. 调用 `graphrag_service.retrieve_context()` 构建查询计划。
4. 基于问题、实体、子查询生成 Embedding，查询 ChromaDB 获得候选片段。
5. 根据相似度、查询权重、实体命中和章节信息综合排序。
6. 扩展命中片段的前后相邻片段，减少分块导致的上下文割裂。
7. 对关系型问题查询 Neo4j 多跳路径，对普通问题筛选支持性图谱证据。
8. 构建系统 Prompt，将参考资料、关系路径和对话历史传给大模型。
9. 生成回答，保存助手消息、引用来源、图谱证据、模型名、Token 消耗和响应时间。

当最大相似度低于阈值且没有可用关系路径时，系统不会强行生成答案，而是返回“知识库中暂无相关信息”的兜底提示，减少无依据编造。

### 5.4 GraphRAG 增强检索

项目在传统向量检索基础上增加了知识图谱增强能力。`graphrag_service.py` 会根据问题中的关键词判断查询模式：

| 查询模式 | 触发场景 | 处理方式 |
|---|---|---|
| fact | 普通事实问答 | 以向量召回为主，补充相关实体 |
| path | 关系、路径、影响、依赖类问题 | 从 Neo4j 查询实体间多跳路径 |
| integration | 综合、对比、跨文档类问题 | 扩大检索 Top-K，召回多个相关片段并整合 |

该设计改善了普通 RAG 在复杂查询中只召回直接片段、遗漏中间概念的问题。系统还将文档分块与图谱实体建立关联，使回答可以同时利用文本证据和结构化关系证据。

### 5.5 知识图谱可视化与维护

知识图谱前端页面基于 ECharts 力导向图实现，支持节点拖拽、缩放、平移、关键词搜索、实体类型颜色区分和节点详情查看。管理员可以手动新增实体、添加关系、删除实体及其关联关系，从而修正自动抽取结果。

后端图谱模块提供图谱概览、关键词搜索、实体维护、关系维护等接口。Neo4j 中保存的实体关系既服务于图谱展示，也参与问答检索阶段的关系路径推理。

### 5.6 模型管理与多供应商支持

系统将模型供应商、对话模型和 Embedding 模型配置存储在 MySQL 中，支持动态维护。管理员可以在模型管理页面新增供应商、编辑 API 地址和 API Key、测试连通性，并为不同模型类型设置默认模型。

模型调用层支持 OpenAI 兼容接口，默认预置 DeepSeek 供应商和 `deepseek-chat`、`deepseek-reasoner`、`deepseek-embedding` 等模型配置。对话模型用于生成回答和知识抽取，Embedding 模型用于文档向量化和问题向量化。API Key 使用加密方式存储，避免明文暴露在数据库中。

### 5.7 系统配置与统计

系统配置表 `system_config` 保存运行时参数，管理员可通过配置页面调整：

| 参数 | 默认值 | 说明 |
|---|---:|---|
| temperature | 0.7 | 大模型生成温度 |
| top_p | 0.9 | 核采样参数 |
| max_tokens | 2048 | 最大生成长度 |
| top_k | 5 | 向量检索返回数量 |
| similarity_threshold | 0.6 | 相似度阈值 |
| chunk_size | 512 | 文本分块大小 |
| chunk_overlap | 128 | 分块重叠长度 |
| kg_enabled | true | 是否启用知识图谱增强 |
| history_rounds | 5 | 多轮对话历史轮数 |

系统还通过 `token_usage` 表记录问答和文档解析中的 Token 消耗，便于后续统计成本、分析模型使用情况。

## 六、数据库设计

### 6.1 MySQL 业务表

MySQL 初始化脚本位于 `docker/mysql/init.sql`，主要表结构如下：

| 表名 | 作用 |
|---|---|
| users | 用户账号、角色、状态和创建时间 |
| documents | 文档元数据、文件类型、解析状态、文件路径、上传用户 |
| conversations | 用户会话 |
| messages | 会话消息、引用来源、图谱证据、模型和 Token 信息 |
| system_config | 系统运行参数 |
| model_providers | 模型供应商、API 地址、加密 API Key |
| model_configs | 对话模型和 Embedding 模型配置 |
| model_presets | 模型预设和 Prompt 参数 |
| token_usage | 问答和文档解析 Token 消耗记录 |

### 6.2 ChromaDB 向量数据

ChromaDB 保存文档分块的向量和元数据。每个 chunk 的元数据包括文档 ID、文件名、分块序号、章节路径、前一块 ID、后一块 ID、起止字符位置等。检索阶段返回片段文本、相似度和元数据，用于构建回答上下文和引用来源。

### 6.3 Neo4j 图谱数据

Neo4j 保存两类图谱信息：

1. 文档结构图：记录文档与分块之间的结构关系，支持上下文扩展和来源定位。
2. 知识实体图：记录从文档中抽取的实体、实体类型、描述、所属文档，以及实体之间的关系。

图谱数据用于管理员可视化查看，也用于问答阶段的多跳路径推理和支持性证据筛选。

## 八、部署操作教程

本项目推荐使用 Docker Compose 部署。Docker Compose 会一次性启动前端、后端、MySQL、Redis、Neo4j 和 ChromaDB，避免手动安装多套运行环境。以下步骤以 Windows 本地部署为主，同时给出 Linux 服务器部署命令。

### 8.1 部署前准备

#### 8.1.1 硬件与系统要求

| 项目 | 最低要求 | 建议配置 |
|---|---|---|
| CPU | 2 核 | 4 核及以上 |
| 内存 | 4 GB | 8 GB 及以上 |
| 磁盘 | 10 GB 可用空间 | 20 GB 及以上，知识库文档较多时继续增加 |

| 网络 | 可访问模型 API 和 Docker 镜像仓库 | 网络稳定，服务器部署时开放必要端口 |

如果在 Windows 上部署，需要提前安装并启动 Docker Desktop；如果在 Linux 服务器上部署，需要安装 Docker Engine 和 Docker Compose Plugin。

#### 8.1.2 必备软件

|---|---|---|

### 8.7 系统初始化使用流程

1. 登录前端：访问 `http://localhost:15173`，使用 `admin/admin123` 登录。
2. 修改默认密码：进入用户或个人资料相关页面，替换默认管理员密码。
3. 配置模型供应商：进入“模型管理”，检查 DeepSeek 供应商 API 地址和 API Key。
4. 测试模型连通性：点击供应商或模型的“测试”按钮，确认 API Key 可用。
5. 检查默认模型：确认对话模型和 Embedding 模型均有一个启用的默认配置。
6. 上传测试文档：进入“知识库管理”，上传 PDF、DOCX、TXT、MD、XLSX 或 CSV 文件。
7. 等待解析完成：文档状态从 `pending`、`parsing` 变为 `completed` 后，才会参与检索。
8. 查看知识图谱：进入“知识图谱”，检查是否抽取出实体和关系。
9. 进行问答测试：进入“智能问答”，输入与文档相关的问题，检查回答和引用来源。
10. 调整检索参数：如果召回不足，可在“系统配置”中调整 `top_k`、`similarity_threshold`、`chunk_size` 等参数。

### 8.11 常见部署故障处理

#### 8.11.1 前端无法访问

排查步骤：

```bash
docker compose ps
docker compose logs rag-frontend --tail 100
```

常见原因：

| 现象 | 原因 | 处理 |
|---|---|---|
| 浏览器打不开 `15173` | 前端容器未启动 | 查看 `rag-frontend` 日志并重启 |
| 页面打开但接口报错 | 后端未启动或代理异常 | 检查 `rag-backend` 状态 |
| 端口被占用 | 本机已有服务使用 15173 | 修改 `.env` 中 `FRONTEND_PORT` |

#### 8.11.2 后端启动失败

排查命令：

```bash
docker compose logs rag-backend --tail 200
```

常见原因：

| 原因 | 处理方式 |
|---|---|
| MySQL 未就绪 | 等待 MySQL healthy 后重启后端：`docker compose restart rag-backend` |
| `.env` 配置错误 | 检查数据库、Redis、Neo4j 密码是否一致 |
| Python 依赖安装失败 | 检查网络，重新执行 `docker compose build rag-backend` |

#### 8.11.3 MySQL 初始化失败

排查命令：

```bash
docker compose logs rag-mysql --tail 200
```

处理建议：

1. 检查 `MYSQL_ROOT_PASSWORD`、`MYSQL_DATABASE`、`MYSQL_USER`、`MYSQL_PASSWORD` 是否为空。
2. 确认 `docker/mysql/init.sql` 存在。
3. 如果已经生成过错误的数据卷，且不需要保留旧数据，可先备份再删除卷重建。

注意：删除卷会清空数据库，执行前必须确认数据可丢弃。

#### 8.11.4 文档一直处于 `failed`

排查步骤：

```bash
docker compose logs rag-backend --tail 200
```

常见原因：

| 原因 | 处理 |
|---|---|
| 文件为扫描版 PDF | 增加 OCR 或改用可复制文本文件 |
| 文件加密或损坏 | 使用正常文件重新上传 |
| API Key 不可用 | 到模型管理页面更新 API Key 并测试 |
| Embedding 模型调用失败 | 检查默认 Embedding 模型配置 |
| Neo4j 暂不可用 | 检查 `rag-neo4j` 状态并重启 |

#### 8.11.5 问答返回“知识库中暂无相关信息”

处理建议：

1. 确认相关文档状态为 `completed`。
2. 降低 `similarity_threshold`，例如从 `0.6` 调整为 `0.3`。
3. 增大 `top_k`，例如从 `5` 调整为 `8` 或 `10`。
4. 换一种更接近文档表述的问法。
5. 如果刚切换 Embedding 模型，需要重新解析已有文档。

#### 8.11.6 模型连通性测试失败

排查项：

1. API Key 是否填写正确。
2. API Base URL 是否为 OpenAI 兼容接口地址。
3. 服务器是否能访问外网。
4. 模型名称是否真实存在。
5. 账户余额或额度是否充足。

可从后端容器内检查网络：

```bash
docker exec -it rag-backend sh
```

进入容器后可使用 Python 或其他可用工具测试目标地址是否可访问。

### 8.12 生产环境部署建议

课程设计或本地演示可直接使用当前 Compose 配置。若部署到真实服务器，建议进一步调整：

1. 前端不要使用 Vite dev server，建议执行 `npm run build` 后由 Nginx 提供静态文件。
2. 后端建议使用多 worker 或进程管理方案，并配置访问日志。
3. CORS 不建议使用 `*`，应限制为实际前端域名。
4. MySQL、Redis、Neo4j、ChromaDB 不建议直接暴露公网端口。
5. 使用 Nginx 反向代理前端和后端，并配置 HTTPS。
6. 配置系统防火墙，只开放 `80`、`443` 或必要管理端口。
7. 建立定期备份任务，至少覆盖 MySQL、Neo4j、ChromaDB 和上传文件。
8. 将 `.env` 文件权限限制为部署用户可读，避免密钥泄露。

3. 长文档上下文处理能力：如长文档后半部分信息检索、跨章节综合和废止声明追踪。
4. 效率与扩展性：如专业术语精准检索、发文号体系跨类检索、领域内外信息区分和 Top-K 截断偏差。

通过这些测试可以评估系统在真实知识库场景下的召回完整性、答案准确性和跨文档综合能力。

### 9.3 部署验收标准

部署完成后应满足以下验收条件：

1. `docker compose ps` 中各服务均处于正常运行状态。
2. 浏览器可访问 `http://localhost:15173` 并完成登录。
3. `http://localhost:18080/health` 返回 `{"status":"ok"}`。
4. `http://localhost:18080/docs` 可打开接口文档。
5. 上传测试文档后，文档状态可变为 `completed`。
6. 基于已上传文档提问时，系统能够返回答案和引用来源。
7. Neo4j 图谱页面可显示或搜索到抽取实体。

## 十、运行维护与安全建议

### 10.1 日常维护

1. 定期查看后端日志，排查文档解析失败、模型调用失败和数据库连接异常。
2. 定期备份 MySQL、ChromaDB、Neo4j 和上传文件。
3. 切换 Embedding 模型后，应重新解析已有文档，避免新旧向量空间不一致。
4. 根据知识库规模调整 `top_k`、`similarity_threshold`、`chunk_size` 和 `chunk_overlap`。
5. 对失败文档检查文件格式、是否加密、是否为扫描版 PDF、文本是否可抽取。

### 10.2 安全建议

1. 生产环境必须修改默认管理员密码。
2. 不要使用默认的 `JWT_SECRET_KEY` 和 `ENCRYPTION_KEY`。
3. 数据库、Redis、Neo4j 密码应使用强密码，并避免将端口直接暴露到公网。
4. CORS 当前允许所有来源，生产环境建议限制为实际前端域名。
5. API Key 应通过模型管理页面加密保存，避免硬编码到源码或公开文档。
6. 建议在服务器前置 Nginx 或网关，启用 HTTPS、访问日志和基础限流。

## 十一、项目问题与优化方向

结合当前项目实现和测试用例，后续可从以下方向优化：

| 优化方向 | 说明 |
|---|---|
| 异步任务队列 | 当前文档解析以后台任务为主，后续可接入 Celery + Redis 实现更可靠的任务调度、重试和进度回传 |
| OCR 能力 | 对扫描版 PDF 增加 OCR 支持，提升不可复制文本的解析能力 |
| 检索重排序 | 在向量召回后增加 reranker，提高专业问题的片段排序质量 |
| 图谱抽取质量 | 增加实体归一化、关系去重、别名合并和人工审核流程 |
| 权限隔离 | 增加知识库级别、文档级别的访问权限，支持多部门隔离 |
| 生产构建 | 前端当前容器以 Vite dev server 运行，生产环境建议改为静态构建并由 Nginx 提供服务 |
| 监控告警 | 增加服务健康检查、资源占用、模型调用耗时和错误率监控 |
| 自动化测试 | 增加后端 API 测试、前端组件测试和端到端测试脚本 |

## 十二、总结

本项目完成了一套完整的 RAG 智能知识库问答系统，从需求分析、架构设计、后端接口、前端页面、多数据库存储、知识图谱增强、模型管理到 Docker 部署均形成了可运行闭环。系统不仅支持传统文档向量检索问答，还引入 Neo4j 知识图谱用于实体关系表达和多跳路径推理，提升了复杂查询、跨文档综合和来源可追溯能力。

在部署层面，项目通过 Docker Compose 将前端、后端、MySQL、Redis、Neo4j、ChromaDB 统一编排，降低了环境搭建和服务启动难度。管理员可通过图形界面维护文档、模型、用户和系统参数，普通用户可直接基于私有知识库进行多轮问答。整体来看，系统具备较好的课程设计完整性、工程可运行性和后续扩展空间。
