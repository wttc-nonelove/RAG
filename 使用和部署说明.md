# RAG 智能知识库问答系统使用和部署说明

## 1. 文档说明

本文档用于说明 RAG 智能知识库问答系统的部署、初始化和使用方法。系统采用 Docker Compose 部署，包含前端、后端、MySQL、Redis、Neo4j 和 ChromaDB 等服务。按照本文档操作，可完成从代码获取、环境配置、服务启动、部署验证到系统使用的完整流程。

项目克隆地址：

```text
https://github.com/wttc-nonelove/RAG.git
```

## 2. 系统简介

本系统是一个基于 RAG（Retrieval-Augmented Generation，检索增强生成）架构的私有知识库智能问答系统。管理员可以上传企业制度、政策法规、技术文档等资料，系统会自动完成文档解析、文本分块、向量化、知识图谱抽取和入库。普通用户可以基于已导入的知识库进行智能问答，系统回答时会结合文档片段、知识图谱证据和多轮对话历史，并展示引用来源。

系统主要功能包括：

| 功能模块 | 说明 |
|---|---|
| 登录认证 | 用户登录、JWT 鉴权、角色权限控制 |
| 智能问答 | 基于知识库的多轮问答、引用来源展示、模型选择 |
| 知识库管理 | 文档上传、解析、预览、删除、重新解析 |
| 知识图谱 | 实体关系抽取、图谱可视化、实体和关系维护 |
| 模型管理 | 模型供应商、对话模型、Embedding 模型和 Prompt 管理 |
| 用户管理 | 创建用户、编辑用户、角色分配、状态管理 |
| 问答历史 | 查看个人历史和管理员查看全量历史 |
| 系统配置 | 调整检索参数、生成参数、分块参数和历史轮数 |

## 3. 技术架构

系统采用前后端分离和多数据库协同架构。

| 层级 | 技术 | 作用 |
|---|---|---|
| 前端 | Vue 3、Vite、Element Plus、Pinia、ECharts | 页面展示、后台管理、图谱可视化 |
| 后端 | Python 3.11、FastAPI、SQLAlchemy、Pydantic | API 服务、业务逻辑、数据访问 |
| 关系数据库 | MySQL 8.0 | 用户、文档、会话、消息、模型和配置数据 |
| 向量数据库 | ChromaDB | 文档分块向量存储和相似度检索 |
| 图数据库 | Neo4j 4.4 | 实体关系、知识图谱和多跳路径查询 |
| 缓存 | Redis 7 | 缓存和后续异步任务扩展 |
| 大模型接口 | DeepSeek API 或 OpenAI 兼容接口 | 对话生成、Embedding、实体关系抽取 |
| 部署 | Docker Compose | 多服务统一编排 |

部署后的服务关系如下：

```text
浏览器
  |
  v
rag-frontend Vue 前端 :15173
  |
  v
rag-backend FastAPI 后端 :18080
  |--------- MySQL :13307
  |--------- Redis :16379
  |--------- ChromaDB :18001
  |--------- Neo4j HTTP :17474 / Bolt :17687
  |
  v
DeepSeek / OpenAI 兼容模型 API
```

## 4. 部署前准备

### 4.1 硬件要求

| 项目 | 最低要求 | 建议配置 |
|---|---|---|
| CPU | 2 核 | 4 核及以上 |
| 内存 | 4 GB | 8 GB 及以上 |
| 磁盘 | 10 GB 可用空间 | 20 GB 及以上 |
| 网络 | 可访问 GitHub、Docker 镜像仓库和模型 API | 网络稳定 |

如果知识库文件较多，ChromaDB、Neo4j、MySQL 和上传文件会持续占用磁盘空间，建议预留充足空间。

### 4.2 软件要求

| 软件 | 用途 | 验证命令 |
|---|---|---|
| Git | 克隆项目代码 | `git --version` |
| Docker | 构建和运行容器 | `docker --version` |
| Docker Compose | 编排多容器服务 | `docker compose version` |
| 浏览器 | 访问系统页面 | Chrome、Edge、Firefox 均可 |

Windows 环境建议安装 Docker Desktop；Linux 服务器环境建议安装 Docker Engine 和 Docker Compose Plugin。

Windows PowerShell 中执行：

```powershell
git --version
docker --version
docker compose version
```

Linux 终端中执行：

```bash
git --version
docker --version
docker compose version
```

如果 Docker 命令无法执行，需要先启动 Docker Desktop 或检查 Docker 服务是否正常运行。

### 4.3 Docker Desktop 准备

Windows 本地部署时，建议进行以下检查：

1. 打开 Docker Desktop。
2. 确认状态为 `Engine running`。
3. 进入 `Settings`，在 `Resources` 中分配至少 4 GB 内存。
4. 如果项目放在 `D:`、`E:`、`F:` 等非系统盘，确认 Docker Desktop 可以访问该路径。
5. 如镜像下载速度慢，可配置可用的 Docker 镜像加速源。

### 4.4 模型 API Key 准备

系统默认使用 DeepSeek API，并兼容 OpenAI 风格接口。部署前需要准备：

1. DeepSeek API Key 或其他 OpenAI 兼容服务的 API Key。
2. API Base URL，例如 DeepSeek 默认为 `https://api.deepseek.com/v1`。
3. 可用的账户余额或调用额度。

如果未配置真实 API Key，系统容器仍可以启动，但模型连通性测试、文档知识抽取和问答生成可能无法正常完成。

### 4.5 默认端口

部署前需要确认以下端口未被占用：

| 宿主机端口 | 服务 | 用途 |
|---:|---|---|
| 15173 | rag-frontend | 前端访问入口 |
| 18080 | rag-backend | 后端 API 和 Swagger 文档 |
| 13307 | rag-mysql | MySQL 数据库 |
| 16379 | rag-redis | Redis 缓存 |
| 17474 | rag-neo4j | Neo4j 浏览器控制台 |
| 17687 | rag-neo4j | Neo4j Bolt 连接 |
| 18001 | rag-chromadb | ChromaDB 向量库 |

Windows 检查端口占用：

```powershell
netstat -ano | findstr ":15173"
netstat -ano | findstr ":18080"
netstat -ano | findstr ":13307"
netstat -ano | findstr ":16379"
netstat -ano | findstr ":17474"
netstat -ano | findstr ":17687"
netstat -ano | findstr ":18001"
```

Linux 检查端口占用：

```bash
ss -lntp | grep -E '15173|18080|13307|16379|17474|17687|18001'
```

如果端口已被占用，可在 `docker/.env` 中修改对应端口。

## 6. 配置环境

### 6.1 创建 `.env` 文件

项目提供了 `.env.example` 作为环境变量模板。首次部署时需要复制为 `.env`。

Windows：

```powershell
Copy-Item .env.example .env
```

Linux：

```bash
cp .env.example .env
```

### 6.2 编辑 `.env`

Windows 可使用记事本：

```powershell
notepad .env
```

Linux 可使用 `nano`：

```bash
nano .env
```

推荐配置如下：

```env
# MySQL
MYSQL_ROOT_PASSWORD=rag_root_2026
MYSQL_DATABASE=rag_db
MYSQL_USER=rag_user
MYSQL_PASSWORD=rag_pass_2026
MYSQL_PORT_HOST=13307

# Redis
REDIS_PASSWORD=rag_redis_2026
REDIS_PORT_HOST=16379

# Neo4j
NEO4J_AUTH=neo4j/rag_neo4j_2026
NEO4J_PASSWORD=rag_neo4j_2026
NEO4J_HTTP_PORT=17474
NEO4J_BOLT_PORT=17687

# ChromaDB
CHROMA_PORT_HOST=18001

# Application ports
FRONTEND_PORT=15173
BACKEND_PORT=18080

# JWT
JWT_SECRET_KEY=change_this_to_a_long_random_string

# AES-256 Encryption Key (32 hex chars = 16 bytes)
ENCRYPTION_KEY=0123456789abcdef0123456789abcdef

# DeepSeek API
DEEPSEEK_API_KEY=sk-your-api-key-here
DEEPSEEK_BASE_URL=https://api.deepseek.com/v1
```

### 6.3 环境变量说明

| 变量 | 说明 |
|---|---|
| `MYSQL_ROOT_PASSWORD` | MySQL root 用户密码 |
| `MYSQL_DATABASE` | 初始化数据库名称，默认 `rag_db` |
| `MYSQL_USER` | 后端连接 MySQL 的业务用户 |
| `MYSQL_PASSWORD` | MySQL 业务用户密码 |
| `MYSQL_PORT_HOST` | MySQL 映射到宿主机的端口 |
| `REDIS_PASSWORD` | Redis 访问密码 |
| `REDIS_PORT_HOST` | Redis 映射到宿主机的端口 |
| `NEO4J_AUTH` | Neo4j 初始化账号和密码，格式为 `neo4j/密码` |
| `NEO4J_PASSWORD` | 后端连接 Neo4j 使用的密码，应与 `NEO4J_AUTH` 中的密码一致 |
| `NEO4J_HTTP_PORT` | Neo4j 浏览器控制台端口 |
| `NEO4J_BOLT_PORT` | Neo4j Bolt 连接端口 |
| `CHROMA_PORT_HOST` | ChromaDB 宿主机端口 |
| `FRONTEND_PORT` | 前端页面访问端口 |
| `BACKEND_PORT` | 后端 API 访问端口 |
| `JWT_SECRET_KEY` | JWT 签名密钥，生产环境必须修改 |
| `ENCRYPTION_KEY` | API Key 加密密钥，要求 32 个十六进制字符 |
| `DEEPSEEK_API_KEY` | DeepSeek API Key |
| `DEEPSEEK_BASE_URL` | DeepSeek API 地址 |

生产环境必须修改所有默认密码和密钥，尤其是 `JWT_SECRET_KEY`、`ENCRYPTION_KEY`、数据库密码、Redis 密码、Neo4j 密码和模型 API Key。

## 9. 系统使用说明

### 9.1 登录系统

1. 打开浏览器访问 `http://localhost:15173`。
2. 输入默认管理员账号 `admin` 和密码 `admin123`。
3. 登录成功后进入系统。
4. 管理员可访问工作台、知识库、知识图谱、模型管理、用户管理等功能。
5. 普通用户主要访问智能问答、我的历史和个人资料。

### 9.2 管理员初始化流程

首次部署后建议按以下顺序初始化：

1. 修改默认管理员密码。
2. 进入“模型管理”，检查 DeepSeek 供应商配置。
3. 填写或更新模型 API Key。
4. 测试模型供应商连通性。
5. 确认对话模型有默认模型，例如 `deepseek-chat`。
6. 确认 Embedding 模型有默认模型，例如 `deepseek-embedding`。
7. 进入“系统配置”，检查 `top_k`、`similarity_threshold`、`chunk_size`、`chunk_overlap` 等参数。
8. 进入“知识库管理”，上传测试文档。
9. 等待文档状态变为 `completed`。
10. 进入“智能问答”，测试知识库问答效果。

### 9.3 智能问答

使用步骤：

1. 点击左侧“智能问答”菜单。
2. 点击“新建对话”或选择已有会话。
3. 在输入框中输入问题。
4. 可选：选择对话模型。
5. 点击发送。
6. 查看 AI 回答、引用来源和相关证据。

使用建议：

1. 问题应尽量明确，包含文档中的关键词、制度名称或业务对象。
2. 如果回答提示知识库中暂无相关信息，先确认相关文档是否解析完成。
3. 如果是连续追问，系统会携带最近若干轮历史上下文。

### 9.4 知识库管理

管理员可在“知识库管理”页面维护文档。

支持的文档格式包括：

```text
PDF、DOCX、TXT、MD、XLSX、XLS、CSV
```

上传流程：

1. 点击“上传文档”。
2. 选择一个或多个文件。
3. 可填写标签。
4. 点击开始上传。
5. 等待文档解析。

文档状态说明：

| 状态 | 说明 |
|---|---|
| `pending` | 已上传，等待解析 |
| `parsing` | 正在解析、分块、向量化和抽取图谱 |
| `completed` | 解析完成，可用于问答 |
| `failed` | 解析失败，可查看日志或重新解析 |

常用操作：

| 操作 | 说明 |
|---|---|
| 预览 | 查看文档抽取后的文本内容 |
| 重新解析 | 重新执行文本分块、向量化和图谱抽取 |
| 删除 | 删除文档记录、文件、向量和图谱关联 |

### 9.5 知识图谱

知识图谱页面用于查看和维护实体关系。

功能包括：

1. 查看全量图谱。
2. 按关键词搜索实体和关系。
3. 拖拽、缩放和平移图谱。
4. 新增实体。
5. 新增关系。
6. 删除实体及其关联关系。

图谱数据主要来自文档解析阶段的大模型实体关系抽取，也可以由管理员手动维护。

### 9.6 模型管理

模型管理包括供应商、对话模型、Embedding 模型和 Prompt 模板。

供应商配置：

1. 进入“模型管理”。
2. 在“模型供应商”中新增或编辑供应商。
3. 填写供应商名称、API 地址和 API Key。
4. 点击测试，确认连通性。

对话模型配置：

1. 切换到“对话模型”。
2. 新增模型配置。
3. 选择供应商。
4. 填写模型名称，例如 `deepseek-chat`。
5. 可填写系统提示词。
6. 设置为默认模型。

Embedding 模型配置：

1. 切换到“Embedding 模型”。
2. 新增模型配置。
3. 填写模型名称和向量维度。
4. 设置为默认模型。

注意：切换 Embedding 模型后，已有文档向量仍由旧模型生成，应重新解析文档以保持向量空间一致。

### 9.7 用户管理

管理员可在“用户管理”页面执行：

1. 新增用户。
2. 设置用户角色为 `admin` 或 `user`。
3. 启用或禁用用户。
4. 修改用户信息。
5. 删除用户。

普通用户只能访问问答和个人相关页面，不能访问管理员管理页面。

### 9.8 问答历史

问答历史用于追踪系统使用情况。

管理员可查看所有用户的问答历史和详情；普通用户可查看自己的历史会话。历史详情中包含用户问题、AI 回答、引用来源、模型名称和 Token 消耗等信息。

### 9.9 系统配置

管理员可在“系统配置”页面调整 RAG 运行参数。

| 参数 | 默认值 | 说明 |
|---|---:|---|
| `temperature` | 0.7 | 生成温度，越高越随机 |
| `top_p` | 0.9 | 核采样参数 |
| `max_tokens` | 2048 | 最大生成长度 |
| `top_k` | 5 | 向量检索返回片段数 |
| `similarity_threshold` | 0.6 | 相似度阈值 |
| `chunk_size` | 512 | 文本分块大小 |
| `chunk_overlap` | 128 | 分块重叠长度 |
| `kg_enabled` | true | 是否启用知识图谱增强 |
| `history_rounds` | 5 | 多轮对话历史轮数 |

如果检索结果过少，可适当降低 `similarity_threshold` 或增大 `top_k`。如果回答上下文不完整，可适当增大 `chunk_size` 或 `chunk_overlap`，修改后建议重新解析文档。


Linux：

```bash
cd /opt/RAG/docker

```

### 12.2 MySQL 恢复

```bash
docker exec -i rag-mysql mysql -u root -p rag_db < backup.sql
```

### 12.3 数据卷注意事项

项目使用 Docker volume 保存 MySQL、Redis、Neo4j、ChromaDB 和文件存储数据。

查看数据卷：

```bash
docker volume ls
```

不要在生产环境随意执行：

```bash
docker compose down -v
```

该命令会删除数据卷，导致数据库、向量库、图谱库和上传文件数据丢失。

## 13. 常见问题处理

### 13.1 前端页面无法访问

排查命令：

```bash
docker compose ps
docker compose logs rag-frontend --tail 100
```

处理方法：

1. 确认 `rag-frontend` 容器处于 `Up` 状态。
2. 确认访问地址为 `http://localhost:15173`。
3. 检查 `FRONTEND_PORT` 是否被占用。
4. 修改端口后重新执行 `docker compose up -d`。

### 13.2 后端接口无法访问

排查命令：

```bash
docker compose ps
docker compose logs rag-backend --tail 200
```

处理方法：

1. 确认 MySQL 和 Redis 已正常启动。
2. 检查 `.env` 中数据库、Redis、Neo4j 密码是否一致。
3. 访问 `http://localhost:18080/health` 检查健康状态。
4. 必要时执行 `docker compose restart rag-backend`。

### 13.3 文档解析失败

常见原因：

| 原因 | 处理 |
|---|---|
| 文件损坏或加密 | 更换正常文件 |
| PDF 为扫描件 | 当前文本抽取可能失败，可改用可复制文本文件 |
| API Key 无效 | 更新模型供应商 API Key 并测试 |
| Embedding 模型配置错误 | 检查默认 Embedding 模型 |
| Neo4j 服务异常 | 检查 `rag-neo4j` 容器日志 |

查看后端日志：

```bash
docker compose logs rag-backend --tail 200
```

### 13.4 问答提示“知识库中暂无相关信息”

处理方法：

1. 确认相关文档状态为 `completed`。
2. 降低 `similarity_threshold`，例如从 `0.6` 调整为 `0.3`。
3. 增大 `top_k`，例如从 `5` 调整为 `8` 或 `10`。
4. 使用更接近文档原文的关键词提问。
5. 如果切换过 Embedding 模型，需要重新解析文档。

### 13.5 模型连通性测试失败

排查项：

1. API Key 是否正确。
2. API Base URL 是否正确。
3. 模型名称是否存在。
4. 账户是否有余额或额度。
5. 当前网络是否能访问模型服务。

### 13.6 MySQL 初始化未生效

原因通常是 MySQL 数据卷已经存在。`init.sql` 只会在数据库首次初始化时执行。

如果确认旧数据可以删除，可执行：

```bash
docker compose down
docker volume ls
```

找到对应 MySQL 数据卷后再谨慎删除。删除数据卷会清空数据库，执行前应先备份。

## 14. 生产环境建议

本项目当前配置适合课程设计、本地演示和测试环境。若部署到生产服务器，建议进行以下调整：

1. 修改所有默认账号、密码和密钥。
2. 前端建议执行生产构建后由 Nginx 提供静态文件，不建议直接使用 Vite dev server。
3. 后端建议增加进程管理、访问日志和错误日志收集。
4. CORS 不建议配置为 `*`，应限制为实际前端域名。
5. MySQL、Redis、Neo4j、ChromaDB 不建议直接暴露到公网。
6. 使用 Nginx 或网关反向代理前端和后端，并启用 HTTPS。
7. 配置防火墙，只开放必要端口。
8. 建立定期备份机制。
9. 对 `.env` 文件设置严格权限，避免 API Key 和数据库密码泄露。
