#### 2.4.2 路由守卫逻辑

```
用户访问任意路径（router.beforeEach）
  → 检查 localStorage.getItem('access_token')
    → Token 不存在且目标不是 /login → 重定向到 /login
    → Token 存在：
      → 解析 JWT payload（jwt_decode），提取 user_id 和 role
      → 检查 Token 是否过期（exp * 1000 < Date.now()）
        → 已过期 → 清除 Token → 重定向 /login
      → 校验角色权限：
        → role='admin' 试图访问 /qa 或 /myhistory → 重定向 /dashboard
        → role='user' 试图访问管理端路由 → 重定向 /qa
        → 角色匹配 → 放行
  → 访问 /login 且已有有效 Token → 按角色跳转默认页
```

#### 2.4.3 路由配置实现

前端路由配置文件 `frontend/src/router/index.js`：

```javascript
// 路由定义核心逻辑
const routes = [
  { path: '/login', component: Login, meta: { guest: true } },
  { path: '/dashboard', component: Dashboard, meta: { role: 'admin' } },
  { path: '/knowledge', component: Knowledge, meta: { role: 'admin' } },
  { path: '/graph', component: Graph, meta: { role: 'admin' } },
  { path: '/users', component: Users, meta: { role: 'admin' } },
  { path: '/history', component: History, meta: { role: 'admin' } },
  { path: '/model', component: Model, meta: { role: 'admin' } },
  { path: '/config', component: Config, meta: { role: 'admin' } },
  { path: '/qa', component: QA, meta: { role: 'user' } },
  { path: '/myhistory', component: MyHistory, meta: { role: 'user' } },
  { path: '/:pathMatch(.*)*', redirect: '/login' },
];

// 导航守卫
router.beforeEach((to, from, next) => {
  const token = localStorage.getItem('access_token');
  if (to.meta.guest) {
    if (token && !isTokenExpired(token)) {
      const payload = parseJwt(token);
      return next(payload.role === 'admin' ? '/dashboard' : '/qa');
    }
    return next();
  }
  if (!token) return next('/login');
  if (isTokenExpired(token)) {
    localStorage.removeItem('access_token');
    return next('/login');
  }
  const payload = parseJwt(token);
  if (to.meta.role && to.meta.role !== payload.role) {
    return next(payload.role === 'admin' ? '/dashboard' : '/qa');
  }
  next();
});
```

#### 2.4.4 前端 Axios 拦截器设计

| 拦截器 | 阶段 | 逻辑 |
|--------|------|------|
| 请求拦截器 | 发送前 | 从 localStorage 读取 access_token → 注入 `Authorization: Bearer <token>` 头；若请求为 FormData 则自动设置 `Content-Type: multipart/form-data` |
| 响应拦截器 | 成功 | 提取 `response.data`，若 `code !== 200` 则统一弹窗提示 message |
| 响应拦截器 | 401 错误 | 清除 localStorage Token → 重定向 `/login`，提示"登录已过期，请重新登录" |
| 响应拦截器 | 5xx 错误 | 统一弹窗提示"服务器异常，请稍后重试" |

Axios 实例配置（`frontend/src/api/client.js`）：

```javascript
const client = axios.create({
  baseURL: '/api/v1',       // Vite proxy → backend:8000
  timeout: 30000,           // 30s 超时
  headers: { 'Content-Type': 'application/json' },
});
```

### 3.3 功能描述

#### 3.3.1 F-01 自然语言提问 + F-02 多轮对话 + F-03 答案溯源 + F-05 用户模型选择

**业务逻辑图**：

```
用户输入问题 (question)
  │
  ├──→ 1. 获取当前会话对话历史（最近 N 轮，默认 5 轮）
  ├──→ 2. 获取用户选择的对话模型及对应 System Prompt 模板
  ├──→ 3. 获取当前 Embedding 模型
  │
  ▼
检索阶段
  ├──→ 4. 问题向量化：
  │       q_vector = embedding_model.encode(question)
  │
  ├──→ 5. ChromaDB 语义检索：
  │       results = chroma_collection.query(
  │           query_embeddings=[q_vector],
  │           n_results=top_k  // 默认 5，可在系统配置中调整
  │       )
  │       返回：[(chunk_text, metadata, similarity_score), ...]
  │
  ├──→ 6. 相似度判断：
  │       if max_similarity < threshold:  // 默认 0.6
  │           return "知识库中暂无相关信息，请尝试更换问题或补充相关文档"
  │           // 流程终止，不调用 LLM
  │       else:
  │           context_chunks = [r for r in results if r.score >= threshold]
  │
  ├──→ 7. Neo4j 知识图谱补充（可选，系统配置开关）：
  │       kg_context = neo4j.query("""
  │           MATCH (e:Entity)
  │           WHERE e.name IN {extracted_terms}
  │           MATCH (e)-[r]->(related)
  │           RETURN e.name, type(r), related.name
  │       """)
  │
  ▼
生成阶段
  ├──→ 8. 组装 Prompt：
  │       system_prompt = prompt_template.format(
  │           context=assemble_context(context_chunks),
  │           history=format_history(recent_messages)
  │       )
  │       user_prompt = f"用户问题：{question}"
  │       full_prompt = system_prompt + "\n\n" + user_prompt
  │
  ├──→ 9. 调用对话模型：
  │       response = llm.chat(
  │           model=selected_model,
  │           messages=[
  │               {"role": "system", "content": system_prompt},
  │               {"role": "user", "content": user_prompt}
  │           ],
  │           temperature=config.temperature,
  │           top_p=config.top_p,
  │           max_tokens=config.max_tokens
  │       )
  │
  ├──→ 10. 解析引用来源：
  │        sources = [{
  │            "doc_name": r.metadata.filename,
  │            "page": r.metadata.page,
  │            "excerpt": r.text[:200]
  │        } for r in context_chunks]
  │
  └──→ 11. 保存问答记录至 MySQL：
          conversation_id, user_id, question, answer, sources, model_name
          返回：{answer, sources, conversation_id, message_id}
```

#### 4.3.1 F-06 文档上传 + F-09 解析状态追踪

**业务逻辑图**：

```
用户上传文档 (PDF/DOCX/TXT)
  │
  ├──→ 1. 文件校验（类型白名单、大小不限）
  ├──→ 2. 存储原始文件到本地磁盘（/data/files/{uuid}.{ext}）
  ├──→ 3. 写入 documents 表（parse_status = "pending"）
  │
  ▼
异步解析任务（FastAPI BackgroundTasks + Redis 队列）
  │
  ├──→ 4. 更新 parse_status = "parsing"
  ├──→ 5. 文本提取：
  │       PDF → pdfplumber.open().pages
  │       DOCX → Document("file.docx").paragraphs
  │       TXT → open("file.txt").read()
  ├──→ 6. 文本清洗：去噪、全角半角转换、空白压缩
  ├──→ 7. 文本分块：
  │       chunk_size = 512 字符（系统配置可调）
  │       chunk_overlap = 128 字符（系统配置可调）
  │       输出：[(chunk_text, page_num, chunk_index), ...]
  │
  ├──→ 8. 向量化 & 入库 ChromaDB：
  │       for chunk in chunks:
  │           vector = embedding_model.encode(chunk.text)
  │           chroma_collection.add(
  │               embedding=vector,
  │               metadata={"doc_id": id, "filename": name,
  │                         "page": chunk.page, "chunk_index": chunk.idx}
  │           )
  │
  ├──→ 9. KG 抽取 & 入库 Neo4j：
  │       full_text = "\n".join([c.text for c in chunks])
  │       entities, relations = llm_ie_extract(full_text)  # LLM 抽取
  │       entities += spacy_ner_extract(full_text)          # spaCy 补充
  │       for entity in entities:
  │           neo4j.merge(Entity{name, type, source_doc_id})
  │       for rel in relations:
  │           neo4j.merge(Relation{head, type, tail})
  │
  └──→ 10. 更新 parse_status = "completed"
         若出错 → parse_status = "failed", 记录 error_message
```

```

        collection.delete(where={"doc_id": doc_id})
  → 3. neo4j_repo.delete_by_doc(doc_id)
        MATCH (e:Entity {source_doc_id: doc_id})
        OPTIONAL MATCH (e)-[r]-()
        DELETE r, e
  → 4. file_store.delete(file_path)
        os.remove(file_path)
  → 5. document_dao.delete(doc_id)
        DELETE FROM documents WHERE id=?
  → 返回 {deleted_id, deleted_filename}
```

|-----------|------|
| **async upload_and_parse(file, tag, user_id) → dict** | |
| | **【方法说明】** 上传文档并触发异步解析 |
| | **【输入参数】** file: UploadFile；tag: str\|None；user_id: int |
| | **【处理过程】** 1. 校验文件扩展名（仅允许 pdf/docx/txt）；2. 生成 UUID 文件名，写入本地磁盘；3. INSERT 到 documents 表（parse_status='pending', version=1）；4. 通过 BackgroundTasks 触发异步解析任务 `parse_document(doc_id)`；5. 返回文档基本信息 |
| | **【输出参数】** `{id, filename, file_type, file_size, parse_status, version}` |
| **async parse_document(doc_id: int) → None** | |
| | **【方法说明】** 异步后台任务 — 解析文档、向量化、KG 抽取 |
| | **【处理过程】** 1. 查询 documents 表获取 file_path 和 file_type；2. 更新 parse_status='parsing'；3. 根据 file_type 选择解析器：PDF → pdfplumber.open(file_path)，DOCX → python-docx Document(file_path)，TXT → open(file_path).read()；4. 文本清洗：去除页眉页脚噪声、全角半角字符转换、多余空白压缩；5. 文本分块：从 system_config 读取 chunk_size（默认 512）和 chunk_overlap（默认 128），逐 chunk 切分，记录 page_num 和 chunk_index；6. 向量化：遍历所有 chunk，调用当前默认 Embedding 模型 encode(chunk_text)，批量写入 ChromaDB collection kb_chunks，metadata 含 doc_id, filename, page, chunk_index；7. KG 抽取：合并全文 → 调用 LLM IE 抽取（prompt: "从以下文本中提取所有实体和关系，以 JSON 格式返回"）→ 调用 spaCy NER 补充实体 → 合并去重 → MERGE 到 Neo4j；8. 更新 parse_status='completed'；9. 若任一步骤异常 → 更新 parse_status='failed'，记录 error_message 到 documents 表 |
| **async update_document(doc_id, file) → dict** | |
| | **【方法说明】** 上传新版本覆盖旧文档 |
| | **【处理过程】** 1. 查询旧文档信息，获取旧 file_path；2. 调用 chroma_repo.delete_by_doc(doc_id) 清理旧向量；3. 调用 neo4j_repo.delete_by_doc(doc_id) 清理旧 KG 数据；4. 删除旧文件 os.remove(old_file_path)；5. 生成新 UUID 文件名，写入新文件到本地磁盘；6. 更新 documents 表：version=version+1, file_size=new_size, file_path=new_path, parse_status='pending', updated_at=now()；7. 通过 BackgroundTasks 触发异步解析任务 parse_document(doc_id)；8. 返回更新后的文档基本信息 |
| **async get_preview(doc_id, page, page_size) → dict** | |
| | **【方法说明】** 获取文档分页预览内容 |
| | **【处理过程】** 1. 查询 documents 表获取 file_path 和 file_type；2. 根据 file_type 选择解析器（pdfplumber/Document/TXT）；3. 按 page 和 page_size 截取文本；4. 返回当前页内容和分页信息 |

#### 10.3.3 F-41 模型预设管理

模型预设（Model Preset）是将**对话模型 + System Prompt 模板 + 生成参数**打包为可复用的配置方案，用户在问答界面可直接选择一个预设来切换模型行为，无需逐个调整参数。

**数据模型**：

```
model_presets 表
├── id (PK)
├── name (VARCHAR 100)          — 预设名称，如 "企业知识库助手"、"英文翻译助手"、"代码审查助手"
├── description (TEXT)          — 预设描述
├── model_config_id (FK → model_configs.id)  — 关联的对话模型
├── system_prompt (TEXT)        — System Prompt 模板（可覆盖关联模型的默认 Prompt）
├── temperature (FLOAT)         — 温度参数覆盖（NULL = 使用系统默认）
├── top_p (FLOAT)               — Top-P 参数覆盖
├── max_tokens (INT)            — Max Tokens 覆盖
├── is_default (BOOLEAN)        — 是否为用户默认预设
├── created_by (FK → users.id)  — 创建者（管理员可创建全局预设，普通用户可创建个人预设）
├── scope (ENUM 'global','personal')  — 预设范围：全局 / 个人
├── created_at, updated_at
```

## 12. 安全设计

### 12.1 认证与授权

#### 12.1.1 JWT Token 认证流程

```
用户登录
  │
  ├── POST /api/v1/auth/login (username + password)
  │     │
  │     ├── UserService.authenticate()
  │     │   ├── 查询 users 表 WHERE username=?
  │     │   ├── bcrypt.verify(password, password_hash)
  │     │   └── 返回 user 对象 或 None
  │     │
  │     ├── 生成 JWT Token:
  │     │   ├── Header: {"alg": "HS256", "typ": "JWT"}
  │     │   ├── Payload: {"sub": user_id, "username": "admin", "role": "admin", "exp": now+86400}
  │     │   └── Signature: HMAC-SHA256(Header.Payload, JWT_SECRET_KEY)
  │     │
  │     └── 返回 {access_token, token_type: "bearer", expires_in: 86400, user: {id, username, role}}
  │
  └── 前端存储:
        └── localStorage.setItem("token", access_token)
              └── Pinia auth store: token 响应式状态
```

#### 12.1.2 请求鉴权流程

```
前端发送请求
  │
  ├── Axios 请求拦截器 (client.js)
  │   ├── 从 localStorage 读取 token
  │   ├── 检查 token 是否存在
  │   │   ├── 不存在 → 放行（登录/注册等公开接口）
  │   │   └── 存在 → 检查是否过期
  │   │       ├── 未过期 → 添加 Authorization: Bearer {token} 头
  │   │       └── 已过期 → 清除 token → 跳转登录页
  │   └── 发送请求
  │
  └── 后端 JWT 依赖注入 (dependencies.py)
      ├── get_current_user(token=Depends(oauth2_scheme))
      │   ├── jwt.decode(token, JWT_SECRET_KEY, algorithms=["HS256"])
      │   ├── 提取 user_id → 查询 users 表
      │   ├── 检查用户 status='active'
      │   └── 返回 user 对象（注入到路由函数参数）
      │
      └── require_admin(current_user=Depends(get_current_user))
          └── 检查 current_user.role == 'admin'，否则返回 403

Axios 响应拦截器:
  ├── 401 → 清除 token → 跳转登录页
  ├── 403 → 提示"无权限"
  └── 其他错误 → 提示 message
```

#### 12.1.3 路由守卫 (Vue Router)

```
router.beforeEach((to, from, next) =>
  ├── to.meta.requiresAuth == false → next()（公开页面）
  ├── token 不存在 → next('/login')
  ├── to.meta.requiresAdmin == true 且 role != 'admin' → next('/qa')
  └── 正常 → next()
)
```

### 12.2 密码安全

| 安全措施 | 实现方式 | 说明 |
|----------|----------|------|
| 密码哈希 | `bcrypt` (passlib) | 使用 bcrypt 算法，自动加盐，工作因子 rounds=12 |
| 密码存储 | `password_hash VARCHAR(255)` | 仅存储哈希值，永不存储明文 |
| 密码验证 | `bcrypt.verify(plain, hash)` | 登录时比对 |
| 密码修改 | 需提供原密码验证 | 防止 token 泄露后被直接改密 |
| 密码重置 | Admin 可强制重置为随机密码 | 随机 12 位（字母+数字+符号） |

### 12.3 API Key 加密存储

| 安全措施 | 实现方式 | 说明 |
|----------|----------|------|
| 加密算法 | AES-256-CBC | Python `cryptography` 库 |
| 密钥来源 | `ENCRYPTION_KEY` 环境变量 | 32 字符密钥，不写入代码/配置文件 |
| 加密时机 | 保存/更新 model_providers 时 | `utils/encryption.py: encrypt(plain_key)` |
| 解密时机 | 调用 LLM API / 测试连通性时 | `utils/encryption.py: decrypt(encrypted_key)` |
| 传输展示 | API 响应中 API Key 脱敏 | `sk-****a1b2`（仅展示前 2 位 + 后 4 位） |

### 12.6 文件上传安全

| 200 | 成功 | 正常响应（业务成功，即使有 fallback 也返回 200） |
| 400 | 请求参数错误 | 缺少必填字段、格式校验失败、文件类型不支持 |
| 401 | 未认证 | Token 缺失、过期、无效 |
| 403 | 无权限 | 角色不满足接口要求（如普通用户访问管理接口） |
| 404 | 资源不存在 | 文档/会话/用户等资源 ID 不存在 |
| 409 | 资源冲突 | 用户名已存在、供应商名称重复 |
| 413 | 文件过大 | 上传文件超过 50MB 限制 |
| 422 | 请求体验证失败 | Pydantic 校验不通过（FastAPI 自动处理） |
| 500 | 服务器内部错误 | 未预期的异常（数据库连接失败、LLM API 超时等） |
| 503 | 服务不可用 | 依赖服务（ChromaDB/Neo4j/LLM API）不可达 |

|--------|------------|---------|----------|-------------|
| 4001 | 400 | 用户名或密码错误 | 登录时用户名不存在或密码不匹配 | 检查输入是否正确 |

### 13.2 错误处理策略

| 策略 | 适用场景 | 实现方式 |
|------|----------|----------|
| **重试** | LLM API 临时故障、网络抖动 | httpx 自动重试（max_retries=3, backoff_factor=0.5），仅对 5xx 和连接错误重试 |
| **降级** | Embedding 服务不可用、图数据库不可用 | 向量检索失败时跳过 RAG 上下文，直接回答"服务暂时不可用"；KG 增强失败时仅使用向量检索结果 |
| **兜底** | 检索结果相似度低于阈值 | 返回"知识库中暂无相关信息，请尝试更换问题或补充相关文档"，不调用 LLM |
| **超时** | LLM API 响应过慢 | httpx timeout=60s，超时后返回"回答生成超时，请稍后重试" |
| **熔断** | 连续失败超过阈值 | 连续 5 次失败后，后续请求直接返回降级响应，60s 后半开尝试恢复 |
| **日志** | 所有异常 | 记录完整 traceback + 请求上下文（user_id, url, params），级别：WARNING（业务异常）/ ERROR（系统异常） |

### 13.3 前端错误处理

```
API 调用出错
  │
  ├── 网络错误 (Axios error.response === undefined)
  │   └── ElMessage.error("网络连接失败，请检查网络")
  │
  ├── 401 未认证
  │   ├── 清除 localStorage token
  │   ├── 清除 Pinia auth store
  │   ├── router.push('/login')
  │   └── ElMessage.warning("登录已过期，请重新登录")
  │
  ├── 403 无权限
  │   └── ElMessage.error("权限不足，无法执行此操作")
  │
  ├── 4xx 客户端错误
  │   └── ElMessage.error(response.data.message)
  │
  ├── 5xx 服务端错误
  │   └── ElMessage.error("服务器繁忙，请稍后重试")
  │
  └── LLM 流式输出中断 (SSE Error)
      └── 保留已输出的文本 + 追加"[回答中断，请重试]"
```

## 附录E ChromaDB Collection Schema

### E.1 Collection 定义

| 属性 | 值 | 说明 |
|------|-----|------|
| **Collection 名称** | `kb_chunks` | 知识库文本块集合 |
| **嵌入模型** | deepseek-embedding（默认 1536 维） | 可切换为其他模型（需重建） |
| **向量维度** | 1536（默认） | 由当前默认 Embedding 模型决定 |
| **距离度量** | cosine（余弦相似度） | ChromaDB 默认 metric |
| **元数据索引** | doc_id, doc_name, chunk_index, page | 用于过滤和溯源 |

### E.2 元数据结构

每条 ChromaDB 记录（document）包含以下元数据：

```json
{
  "id": "chunk_uuid_v4",
  "embedding": [0.023, -0.145, ...],
  "document": "第三章 数据安全策略\n3.1 访问控制\n企业应建立严格的访问控制机制...",
  "metadata": {
    "doc_id": 12,
    "doc_name": "企业安全管理制度 v3.0.pdf",
    "chunk_index": 8,
    "page": 14,
    "total_chunks": 45,
    "file_type": "PDF",
    "tag": "安全规范",
    "chunk_size": 512,
    "created_at": "2026-05-28T10:30:00Z"
  }
}
```

| 元数据字段 | 类型 | 说明 | 用途 |
|-----------|------|------|------|
| doc_id | int | 关联 documents 表主键 | 级联删除、溯源 |
| doc_name | str | 文档名称 | 展示引用来源 |
| chunk_index | int | 当前块序号（从 0 开始） | 定位上下文位置 |
| page | int | 所在页码（PDF 专用，TXT 为 1） | 展示引用页码 |
| total_chunks | int | 该文档总块数 | 进度展示 |
| file_type | str | 文件类型 (PDF/DOCX/TXT) | 统计 |
| tag | str | 文档标签 | 按标签筛选检索范围（可选） |
| chunk_size | int | 分块大小 | 调试/监控 |
| created_at | str | 创建时间 | 时间维度查询 |

### E.3 核心操作接口

| 操作 | 调用方法 | 说明 |
|------|----------|------|
| 获取/创建 Collection | `client.get_or_create_collection("kb_chunks", metadata={"hnsw:space": "cosine"})` | 首次自动创建 |
| 批量添加向量 | `collection.add(ids=[...], embeddings=[...], metadatas=[...], documents=[...])` | 文档解析后批量写入 |
| 向量检索 | `collection.query(query_embeddings=[vector], n_results=top_k, where={"doc_id": 12})` | 支持 metadata 过滤 |
| 按 doc_id 删除 | `collection.delete(where={"doc_id": 12})` | 文档删除时级联清理 |
| 全量重建 | 创建新 collection → 逐文档重新 Embedding → 删除旧 collection → 重命名 | Embedding 模型切换时 |
| 健康检查 | `client.heartbeat()` | 返回心跳时间戳 |

### E.4 检索接口

```
ChromaRepository (dao/chroma_repo.py)
├── query(query_embedding: List[float], top_k: int, doc_id: int = None)
│   → List[{id, document, metadata, distance}]
│   → 按 cosine 距离升序排列（距离越小越相似）
│
├── add_batch(ids: List[str], embeddings: List[List[float]],
│              metadatas: List[dict], documents: List[str]) → None
│
├── delete_by_doc_id(doc_id: int) → int   # 返回删除条数
│
├── count_by_doc_id(doc_id: int) → int    # 返回文档块数
│
├── get_collection_stats() → dict         # {name, count, dimension}
│
└── check_health() → bool                 # heartbeat() + list_collections()
```

### F.2 关系类型定义

| 关系类型 | 方向 | 说明 | 属性 | 示例 |
|----------|------|------|------|------|

### F.3 图操作接口

```
Neo4jRepository (dao/neo4j_repo.py)
├── get_overview() → dict
│   → {entity_count: 156, relation_count: 312, entity_types: [...], top_entities: [...]}
│   → 查询: MATCH (e:Entity) RETURN count(e); MATCH ()-[r]->() RETURN count(r)
│
├── get_entities(page, page_size, type_filter, search) → (items, total)
│   → 查询: MATCH (e:Entity) WHERE ... RETURN e ORDER BY e.name SKIP $skip LIMIT $limit
│
├── create_entity(name, type, doc_id, doc_name, description) → entity
│   → MERGE 避免重复; 生成 entity_id 使用自增序列
│
├── get_entity(entity_id) → entity | None
│   → MATCH (e:Entity {entity_id: $id}) RETURN e
│
├── update_entity(entity_id, updates) → entity
│   → MATCH → SET
│
├── delete_entity(entity_id) → None
│   → MATCH (e:Entity {entity_id: $id}) DETACH DELETE e
│   → DETACH DELETE 自动删除所有关联关系
│
├── get_entity_relations(entity_id) → [{entity, relation}]
│   → MATCH (e:Entity {entity_id: $id})-[r]-(related:Entity) RETURN e, r, related
│
├── create_relation(from_entity_id, to_entity_id, type, weight, doc_id) → relation
│   → MATCH (a:Entity {entity_id: $fid}), (b:Entity {entity_id: $tid})
│   → MERGE (a)-[r:RELATES_TO]->(b) SET r.weight=$w, r.doc_id=$did
│
├── delete_relation(relation_id) → None
│
├── search(keyword, limit=50) → [{entity, score}]
│   → MATCH (e:Entity) WHERE e.name CONTAINS $keyword RETURN e
│   → 支持模糊匹配，按名称排序
│
├── get_subgraph(entity_names: List[str], depth=1) → {nodes, edges}
│   → MATCH path = (e:Entity)-[*1..depth]-(related) WHERE e.name IN $names RETURN path
│   → 用于 RAG 知识图谱增强检索
│
├── delete_by_doc_id(doc_id) → int  # 返回删除节点数
│   → MATCH (e:Entity {doc_id: $did}) DETACH DELETE e
│
└── check_health() → bool
    → 执行 RETURN 1 验证连接
```

### F.4 知识图谱可视化数据格式

前端 GraphCanvas 组件通过以下接口获取图数据：

```
GET /api/v1/graph/overview → {nodes: [...], edges: [...]}

nodes: [{entity_id, name, type, doc_id, doc_name, description, color, size}]
  - color: 由 type 映射（概念=#409EFF, 部门=#67C23A, 角色=#E6A23C, 流程=#F56C6C, 系统=#909399）
```

### F.5 级联删除规则

```
删除文档 (DELETE /api/v1/documents/{id})
  ├──[1] documents 表 DELETE (MySQL)
  ├──[2] kb_chunks collection DELETE where={"doc_id": id} (ChromaDB)
  ├──[3] MATCH (e:Entity {doc_id: id}) DETACH DELETE e (Neo4j)
  └──[4] 删除物理文件 (backend/data/files/)

DETACH DELETE 会级联删除该 Entity 节点的所有关系，
避免孤立的 [:RELATES_TO] / [:INCLUDES] / [:MANAGES] 关系残留。
```
