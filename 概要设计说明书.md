## 1. 引言

### 1.1 编写目的

本文档为"基于RAG架构的智能问答系统"的概要设计说明书，依据《软件需求规格说明书》（v1.9）编制。目的在于：

- 明确系统的整体架构、模块划分及模块间协作关系
- 定义系统接口、数据结构及核心处理流程
- 为后续详细设计和编码实现提供框架性指导
- 作为项目评审、测试验收的技术依据

预期读者：开发团队、课程设计评审教师。

### 1.2 范围

本系统是一个面向企业用户的私有知识库智能问答平台，基于检索增强生成（RAG）架构，支持：

- 管理员：知识库文档管理、用户管理、知识图谱管理、模型与 API 管理、工作台仪表盘、系统配置、全局问答历史查看
- 普通用户：基于知识库的多轮智能问答、个人问答历史查看

本文档覆盖系统的总体设计、接口设计、数据结构设计、运行设计、出错处理设计及安全设计，不涉及具体代码实现细节。

### 1.3 定义与缩略语

| 术语/缩略语 | 说明 |
|-------------|------|
| RAG | 检索增强生成（Retrieval-Augmented Generation），结合信息检索与 LLM 生成的技术范式 |
| LLM | 大语言模型（Large Language Model），本系统默认使用 DeepSeek-V4-Pro |
| Embedding | 文本向量化嵌入，将文本转换为稠密向量表示 |
| ChromaDB | 开源向量数据库，用于存储和检索文档文本块向量 |
| Neo4j | 图数据库，用于存储知识图谱实体及关系 |
| KG | 知识图谱（Knowledge Graph），实体-关系-实体的结构化知识表示 |
| SPA | 单页应用（Single Page Application），前端架构模式 |
| JWT | JSON Web Token，用于无状态用户认证 |
| RESTful API | 符合 REST 风格的 HTTP API 设计 |
| Chunk | 文本分块，将长文档切分为固定长度的文本片段 |
| System Prompt | 系统提示词，定义 LLM 的角色与行为约束 |
| IE | 信息抽取（Information Extraction），从非结构化文本中提取实体和关系 |

### 2.4 核心处理流程

#### 2.4.1 文档解析入库流程

```
用户上传文档 (PDF/DOCX/TXT)
  │
  ├──→ 1. 文件校验（类型白名单、大小不限）
  ├──→ 2. 存储原始文件到本地磁盘（/data/files/{uuid}.{ext}）
  ├──→ 3. 写入 documents 表（parse_status = "pending"）
  │
  ▼
异步解析任务（Celery / BackgroundTasks）
  │
  ├──→ 4. 更新 parse_status = "parsing"
  ├──→ 5. 文本提取：
  │       PDF → pdfplumber.open().pages
  │       DOCX → Document("file.docx").paragraphs
  │       TXT → open("file.txt").read()
  ├──→ 6. 文本清洗：去噪、全角半角转换、空白压缩
  ├──→ 7. 文本分块：
  │       chunk_size = 512 字符（系统配置可调）
  │       chunk_overlap = 128 字符（系统配置可调）
  │       输出：[(chunk_text, page_num, chunk_index), ...]
  │
  ├──→ 8. 向量化 & 入库 ChromaDB：
  │       for chunk in chunks:
  │           vector = embedding_model.encode(chunk.text)
  │           chroma_collection.add(
  │               embedding=vector,
  │               metadata={"doc_id": id, "filename": name,
  │                         "page": chunk.page, "chunk_index": chunk.idx}
  │           )
  │
  ├──→ 9. KG 抽取 & 入库 Neo4j：
  │       full_text = "\n".join([c.text for c in chunks])
  │       entities, relations = llm_ie_extract(full_text)  # LLM 抽取
  │       entities += spacy_ner_extract(full_text)          # spaCy 补充
  │       for entity in entities:
  │           neo4j.merge(Entity{name, type, source_doc_id})
  │       for rel in relations:
  │           neo4j.merge(Relation{head, type, tail})
  │
  └──→ 10. 更新 parse_status = "completed"
         若出错 → parse_status = "failed", 记录 error_message
```

#### 2.4.2 RAG 问答流程

```
用户输入问题 (question)
  │
  ├──→ 1. 获取当前会话对话历史（最近 N 轮，默认 5 轮）
  ├──→ 2. 获取用户选择的对话模型及对应 System Prompt 模板
  ├──→ 3. 获取当前 Embedding 模型
  │
  ▼
检索阶段
  ├──→ 4. 问题向量化：
  │       q_vector = embedding_model.encode(question)
  │
  ├──→ 5. ChromaDB 语义检索：
  │       results = chroma_collection.query(
  │           query_embeddings=[q_vector],
  │           n_results=top_k  // 默认 5，可在系统配置中调整
  │       )
  │       返回：[(chunk_text, metadata, similarity_score), ...]
  │
  ├──→ 6. 相似度判断：
  │       if max_similarity < threshold:  // 默认 0.6
  │           return "知识库中暂无相关信息，请尝试更换问题或补充相关文档"
  │           // 流程终止，不调用 LLM
  │       else:
  │           context_chunks = [r for r in results if r.score >= threshold]
  │
  ├──→ 7. Neo4j 知识图谱补充（可选，系统配置开关）：
  │       kg_context = neo4j.query("""
  │           MATCH (e:Entity)
  │           WHERE e.name IN {extracted_terms}
  │           MATCH (e)-[r]->(related)
  │           RETURN e.name, type(r), related.name
  │       """)
  │
  ▼
生成阶段
  ├──→ 8. 组装 Prompt：
  │       system_prompt = prompt_template.format(
  │           context=assemble_context(context_chunks),
  │           history=format_history(recent_messages)
  │       )
  │       user_prompt = f"用户问题：{question}"
  │       full_prompt = system_prompt + "\n\n" + user_prompt
  │
  ├──→ 9. 调用对话模型：
  │       response = llm.chat(
  │           model=selected_model,
  │           messages=[
  │               {"role": "system", "content": system_prompt},
  │               {"role": "user", "content": user_prompt}
  │           ],
  │           temperature=config.temperature,
  │           top_p=config.top_p,
  │           max_tokens=config.max_tokens
  │       )
  │
  ├──→ 10. 解析引用来源：
  │        sources = [{
  │            "doc_name": r.metadata.filename,
  │            "page": r.metadata.page,
  │            "excerpt": r.text[:200]
  │        } for r in context_chunks]
  │
  └──→ 11. 保存问答记录至 MySQL：
          conversation_id, user_id, question, answer, sources, model_name
          返回：{answer, sources, conversation_id, message_id}
```

#### 2.4.3 模型切换与向量重建流程

```
管理员切换 Embedding 模型
  │
  ├──→ 1. 系统获取新模型的向量维度
  │       new_dim = model.embedding_dimension
  │
  ├──→ 2. 检查与当前 ChromaDB 向量的兼容性：
  │       current_dim = chroma_collection.get_dimension()
  │       if new_dim == current_dim:
  │           直接切换，即时生效 ✓
  │       else:
  │           弹出提示："已有向量 (维度 {current_dim}) 与新模型 (维度 {new_dim}) 不兼容，
  │                      需全量重建向量库"
  │           管理员可选择：
  │           [A] 一键重建向量库
  │           [B] 手动调整目标维度
  │           [C] 取消切换
  │
  ├──→ 3. 若选择 [A] 一键重建：
  │       创建异步任务 → 遍历全部 documents
  │       → 逐个重新解析（chunk → 新模型编码 → 写入新 ChromaDB collection）
  │       → 重建期间，旧 collection 仍可查询
  │       → 全部完成后，切换 collection 别名，删除旧 collection
  │       → 更新 model_configs.is_default = TRUE
  │
  └──→ 4. 若选择 [B] 手动调整：
         管理员输入目标维度值
         → 若等于当前维度，直接切换
         → 若不等于，仍需全量重建（流程同 [A]）
```

#### 2.4.4 文档更新流程

```
管理员选择文档 → 点击"上传新版本"
  │
  ├──→ 1. 选择新文件 → 前端校验类型
  ├──→ 2. 后端接收新文件：
  │       ┌── 查询旧文档 ChromaDB 向量 → 批量删除
  │       └── 查询旧文档 Neo4j 图谱节点/关系 → 批量删除
  │
  ├──→ 3. 存储新文件（覆盖旧文件路径或新 UUID + 更新 file_path）
  ├──→ 4. 更新 documents 表：
  │       version = version + 1
  │       file_size = new_size
  │       parse_status = "pending"
  │       updated_at = now()
  │
  ├──→ 5. 触发文档解析入库流程（同 2.4.1）
  │
  └──→ 6. 历史问答中的旧引用链接保留，但标记为"文档已更新至 v{N}"
```

## 3. 接口设计

### 3.1 前端路由设计

系统共设 **10 个前端路由**：

| 路由路径 | 页面名称 | 对应模块 | 访问角色 | 说明 |
|----------|----------|----------|----------|------|
| `/login` | 登录页 | M5 | 所有用户 | 默认首页，含角色选择和注册入口 |
| `/dashboard` | 工作台 | M6 | 管理员 | 仪表盘统计 |
| `/knowledge` | 知识库管理 | M2 | 管理员 | 文档列表、上传、预览、更新 |
| `/graph` | 知识图谱 | M3 | 管理员 | 图谱可视化、实体/关系管理 |
| `/users` | 用户管理 | M5 | 管理员 | 用户列表、新增、编辑、禁用 |
| `/history` | 全局问答历史 | M4 | 管理员 | 全平台问答记录与统计 |
| `/model` | 模型与 API 管理 | M8 | 管理员 | 供应商/模型/Prompt 管理 |
| `/config` | 系统配置 | M7 | 管理员 | 模型参数、检索参数、分块策略 |
| `/qa` | 智能问答 | M1 | 普通用户 | 多轮对话、模型选择 |
| `/myhistory` | 个人问答历史 | M4 | 普通用户 | 个人历史记录 |

**路由守卫逻辑**：

```
用户访问任意路径
  → 检查 localStorage 中是否有有效 JWT Token
    → 无 Token → 重定向到 /login
    → 有 Token → 解析 payload.role
      → role = "admin" → 仅可访问管理端路由
      → role = "user"  → 仅可访问用户端路由
      → 越权访问 → 重定向至角色默认页
```

### 3.2 RESTful API 设计

API 统一前缀：`/api/v1`

通用规范：
- 请求头：`Authorization: Bearer {JWT_TOKEN}`（除登录/注册外）
- 请求体：`application/json`（文件上传使用 `multipart/form-data`）
- 响应格式：`{"code": 200, "data": {...}, "message": "ok"}`（成功）
- 错误格式：`{"code": 4xx/5xx, "message": "错误描述", "detail": "..."}`

#### 3.2.1 认证接口 (Auth)

| 方法 | 路径 | 说明 | 鉴权 |
|------|------|------|------|
| POST | `/api/v1/auth/login` | 用户登录，返回 JWT Token | 无 |
| POST | `/api/v1/auth/register` | 普通用户自主注册 | 无 |
| GET | `/api/v1/auth/me` | 获取当前用户信息 | JWT |
| PUT | `/api/v1/auth/password` | 修改个人密码 | JWT |

**POST /api/v1/auth/login 请求/响应示例**：

```
请求：
{
  "username": "admin",
  "password": "123456"
}

响应：
{
  "code": 200,
  "data": {
    "access_token": "eyJhbGciOiJIUzI1NiIs...",
    "token_type": "bearer",
    "expires_in": 86400,
    "user": {
      "id": 1,
      "username": "admin",
      "role": "admin"
    }
  }
}
```

#### 3.2.2 文档管理接口 (Documents)

| 方法 | 路径 | 说明 | 鉴权 |
|------|------|------|------|
| GET | `/api/v1/documents` | 文档列表（分页、搜索、筛选） | Admin |
| POST | `/api/v1/documents/upload` | 上传新文档 | Admin |
| GET | `/api/v1/documents/{id}` | 文档详情 | Admin |


**检索接口封装**：
```python
# 查询
results = collection.query(
    query_embeddings=[vector],
    n_results=top_k,
    include=["documents", "metadatas", "distances"]

# 返回：List[Tuple[text, metadata, distance]]

# 按文档 ID 删除
collection.delete(where={"doc_id": doc_id})

new_collection = client.create_collection(
    name=f"kb_chunks_v2_{timestamp}",
```

### 4.3 Neo4j 图数据库设计

| 元素 | 说明 |
|------|------|
| 数据库名 | `neo4j` (默认) |
| 连接方式 | `bolt://localhost:7687` |

**节点标签**：`Entity`

**Entity 属性**：
| 属性 | 类型 | 说明 |
|------|------|------|
| name | STRING | 实体名称（唯一标识） |
| type | STRING | 实体类型（概念/技术/人名/地名/组织/产品/日期/其他） |
| source_doc_id | INTEGER | 来源文档 ID |
| source_doc_name | STRING | 来源文档名称 |
| created_at | DATETIME | 创建时间 |

**关系类型**（模型自由抽取，常见类型如下）：
- `包含` — 整体与部分
- `依赖` — 技术/逻辑依赖
- `属于` — 类别归属
- `等同于` — 同义关系
- `前置于` — 顺序关系
- `关联` — 一般关联

**关系属性**：
| 属性 | 类型 | 说明 |
|------|------|------|
| source_doc_id | INTEGER | 来源文档 ID |
| created_at | DATETIME | 创建时间 |

**索引**：
```cypher
CREATE INDEX entity_name_idx FOR (e:Entity) ON (e.name);
CREATE INDEX entity_type_idx FOR (e:Entity) ON (e.type);
CREATE INDEX entity_source_idx FOR (e:Entity) ON (e.source_doc_id);
```

**文档删除时的级联清理**：
```cypher
MATCH (e:Entity {source_doc_id: $doc_id})
OPTIONAL MATCH (e)-[r]-()
DELETE r, e
```

### 4.4 文件存储设计

| 属性 | 说明 |
|------|------|
| 存储方式 | 本地磁盘 |
| 根目录 | `./data/files/` |
| 文件命名 | `{uuid}.{ext}` （如 `a1b2c3d4.pdf`） |
| 目录结构 | 平铺存储（不分子目录），documents 表 file_path 字段记录完整路径 |

**文件操作规范**：
- 上传：`uuid = str(uuid4())`, `path = f"./data/files/{uuid}.{ext}"`
- 预览：根据 `file_path` 读取文件 → 解析 → 返回文本（分页）
- 删除：`os.remove(file_path)`
- 更新：删除旧文件路径的文件，写入新文件，更新 `file_path`

## 6. 系统出错处理设计

### 6.1 出错信息

| 错误码 | HTTP 状态码 | 错误场景 | 用户提示 |
|--------|------------|----------|----------|
| 1001 | 400 | 用户名或密码错误 | "用户名或密码错误" |
| 1002 | 401 | JWT Token 过期 | "登录已过期，请重新登录" |
| 1003 | 403 | 无权限访问 | "无权限访问该资源" |
| 2001 | 400 | 文件格式不支持 | "仅支持 PDF、DOCX、TXT 格式" |
| 2002 | 404 | 文档不存在 | "文档不存在或已被删除" |
| 2003 | 500 | 文档解析失败 | "文档解析失败：{原因}，请检查文件后重试" |
| 3001 | 400 | 问题内容为空 | "请输入问题" |
| 3002 | 503 | LLM API 调用超时 | "模型服务暂时不可用，请稍后重试" |
| 3003 | 503 | LLM API 调用频率限制 | "请求过于频繁，请稍后重试" |
| 4001 | 400 | 模型连通性测试失败 | "API 连接失败：{原因}，请检查 URL 和 Key" |
| 4002 | 400 | 切换 Embedding 模型维度不兼容 | "向量维度不兼容，需全量重建向量库" |
| 4003 | 409 | 向量重建进行中 | "向量重建任务正在进行中，请等待完成后再操作" |
| 5001 | 500 | 数据库连接失败 | "系统内部错误，请联系管理员" |
| 5002 | 503 | ChromaDB 不可用 | "检索服务暂时不可用" |
| 5003 | 503 | Neo4j 不可用 | "知识图谱服务暂时不可用" |

### 6.2 补救措施

| 错误类型 | 补救策略 |
|----------|----------|
| LLM API 调用失败 | 自动重试 2 次（间隔 1s, 3s），仍失败则返回降级提示；记录错误日志 |
| ChromaDB 检索异常 | 返回兜底提示"检索服务异常"，不阻塞问答流程，降级为仅 LLM 回答 |
| Neo4j 查询异常 | 跳过 KG 增强环节，仅使用向量检索结果，不影响核心问答功能 |
| 文档解析失败 | 标记 parse_status="failed"，记录 error_message，管理员可手动"重新解析" |
| MySQL 写入失败 | 事务回滚，返回明确错误信息，记录完整日志 |
| 向量重建中断 | 支持断点续建，从已完成的文档后继续，不影响正在进行的检索服务 |

### 6.3 日志策略

| 日志级别 | 记录内容 |
|----------|----------|
| INFO | 用户登录/登出、文档上传/删除、问答请求/响应、模型切换、配置变更 |
| WARNING | API 调用重试、检索相似度过低、解析耗时异常（>30s）、磁盘空间不足（<20%） |
| ERROR | API 调用失败、数据库连接失败、文档解析失败、向量重建异常 |
| DEBUG | Prompt 组装详情、检索中间结果（开发调试用，生产环境关闭） |

日志格式：`[时间] [级别] [模块] 用户:{user_id} {消息} {关键参数}`

## 7. 安全设计

### 7.1 认证与授权

| 环节 | 方案 |
|------|------|
| 密码存储 | bcrypt + salt 哈希，不可逆 |
| Token 机制 | JWT (HS256)，过期时间 24h，payload 含 user_id + role |
| Token 刷新 | 前端拦截 401 → 清除 Token → 重定向登录页 |
| 接口鉴权 | FastAPI Depends 依赖注入，解析 Authorization Header Bearer Token |
| 角色鉴权 | 装饰器 `@requires_role("admin")` 校验 payload.role |
| 密码安全 | 最小长度 6 位 |

### 7.2 数据安全

| 环节 | 方案 |
|------|------|
| API Key 存储 | AES-256-CBC 加密存储于 model_providers.api_key_encrypted 字段，密钥从环境变量 `ENCRYPTION_KEY` 读取 |
| 传输安全 | 生产环境启用 HTTPS，开发环境明文 HTTP |
| 敏感信息脱敏 | API Key 在日志和前端仅显示前 4 位 + 后 4 位（如 `sk-a1****b2c3`） |
| CORS | FastAPI CORSMiddleware，开发环境允许 localhost:5173，生产环境限制具体域名 |
| SQL 注入防护 | SQLAlchemy ORM 参数化查询 |
| XSS 防护 | 前端 Vue 模板自动转义，后端返回 JSON |

## 9. 附录

### 9.1 设计决策记录

| # | 决策项 | 决策 | 理由 |
|---|--------|------|------|
| 1 | 前后端分离 | 是 | 前端 Vue 3 SPA + 后端 FastAPI RESTful API，独立开发、独立部署 |
| 2 | 向量数据库 | ChromaDB 本地持久化 | 轻量级，零配置，适合课程设计场景 |
| 3 | 异步任务 | Celery + Redis | 文档解析和向量重建耗时长，需异步非阻塞处理 |
| 4 | API Key 加密 | AES-256-CBC，密钥从环境变量读取 | 安全需求，防止数据库泄露导致 API Key 暴露 |
| 5 | LLM 调用方式 | 统一适配层 + OpenAI SDK 兼容模式 | 支持多供应商切换，无需修改业务代码 |
| 6 | 前端图表 | ECharts | 国产开源，中文文档友好，后台管理场景成熟 |
| 7 | 知识图谱抽取 | 全自动，LLM 为主 + spaCy 辅助 | 减少人工审核成本，课程设计场景合理 |
| 8 | 问答检索兜底 | 相似度 < 0.6 不调用 LLM | 节省 API 成本，避免 LLM 幻觉，用户体验更明确 |
